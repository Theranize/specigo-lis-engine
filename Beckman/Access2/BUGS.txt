BUG-001  Hard-coded DB credentials in IntegrationEngine._createDbPool()
  File   : src/engine/IntegrationEngine.js  (~line 276)
  Detail : host, user, password, database, port, poolSize are all hardcoded
           literal strings ('127.0.0.1', 'admin', 'admin', 'lis_db', ...).
           The .env variables DB_HOST, DB_USER, DB_PASSWORD, DB_NAME are
           completely ignored. The engine always tries 127.0.0.1/admin.
           The validation check `if (!host || !user || !password)` is now
           permanently false (dead code) so bad credentials are never caught.
  Impact : Engine will fail to connect to the real database in any non-dev
           environment. Credentials exposed as plaintext in source code.
  Fix    : Replace hardcoded values with process.env.DB_HOST, DB_USER,
           DB_PASSWORD, DB_NAME, DB_PORT, DB_POOL_SIZE (as it was before
           the regression).

BUG-002  mapping_status / result_status / result_datetime / loinc_id
         not written to lis_results table
  File   : src/db/ResultWriter.js  write() method
  Detail : ParameterMapper produces these fields on every MappedResult:
             mapping_status  ('MAPPED' | 'UNMAPPED')
             result_status   ('F' | 'P' | 'C' | 'X')
             result_datetime (YYYYMMDDHHMMSS)
             loinc_id
             raw_value
           The INSERT SQL in write() only covers 11 columns and omits all
           of these. They are silently dropped and never stored.
           IntegrationAPI._getResults() filters rows by
           `WHERE mapping_status = ?` - this query always returns 0 rows
           because the column is never populated.
  Impact : Dashboard "status" filter is completely broken. mapping_status,
           result_status and result_datetime are invisible in the database.
  Fix    : Add the missing columns to the INSERT statement and params list.

BUG-003  autoOpen undefined causes port-open race condition
  File   : src/transport/SerialPortManager.js  _openPort()
  Detail : IntegrationEngine passes `autoOpen: cfg.connection.autoOpen` to
           SerialPortManager. That field does not exist in access2_config.json
           so cfg.connection.autoOpen is undefined. The serialport library
           defaults autoOpen to TRUE when the option is undefined. This means
           the port opens immediately on `new SerialPort(...)` construction,
           BEFORE the 'open' event listener is attached in the lines that
           follow. If the port opens in the same tick, resolve() is never
           called and _openPort() hangs indefinitely.
  Impact : Engine can hang on startup and never emit 'connected'. Intermittent
           depending on OS port open timing.
  Fix    : Hardcode `autoOpen: false` in _openPort() options regardless of
           config value. Remove autoOpen from the options passed to
           SerialPortManager constructor in IntegrationEngine._initialiseModules().

BUG-004  LIMS API push fails silently with no retry - results lost
  File   : src/db/ResultWriter.js  _sendToLimsApi()
  Detail : The LIMS server push is fire-and-forget. If the server is
           unavailable, the request times out after 10s and is logged as an
           error, but the result is never retried. Results saved to the local
           MySQL DB are never pushed again. There is no queue, no pending-push
           flag in DB, and no alert to operators.
  Impact : Results accumulate in local DB but do not reach the LIMS server
           during network outages. Silent data gap with no visibility.
  Fix    : Either (a) store a push_status column in lis_results and implement
           a periodic retry sweep, or (b) log failed pushes to a separate
           table and expose a retry endpoint in IntegrationAPI.

BUG-005  IntegrationEngine._writeToPort() accesses private _port property
  File   : src/engine/IntegrationEngine.js  _writeToPort()
  Detail : `this._serialManager._port.write(buffer, cb)` directly accesses
           the internal `_port` property of SerialPortManager. This is a
           private implementation detail. If serialport library updates change
           the internal property name, this breaks silently. SerialPortManager
           exposes no public write() method.
  Impact : Fragile coupling. Will break undetected on serialport upgrade.
  Fix    : Add a write(buffer) public method to SerialPortManager that
           delegates to this._port.write(). Use that in _writeToPort().

BUG-006  _upsertMapping() returns HTTP 201 for updates
  File   : IntegrationAPI.js  _upsertMapping()
  Detail : `ok(res, { ... }, 201)` is returned for both INSERT and UPDATE
           (ON DUPLICATE KEY UPDATE). HTTP 201 Created is semantically wrong
           for an update operation which should return 200 OK.
  Fix    : Return 201 only when result.affectedRows === 1 && result.insertId > 0
           (genuine insert). Return 200 for updates.

BUG-007  _getLog() selects sample_id column that is never inserted
  File   : IntegrationAPI.js  _getLog()  /  src/db/ResultWriter.js logSessionEvent()
  Detail : The SELECT in _getLog() includes `l.sample_id`. ResultWriter
           logSessionEvent() never populates a sample_id column - only
           lab_uid, analyzer_uid, session_type, message_code, details, created_at
           are written. sample_id will always be null or cause a column-not-found
           error depending on the table schema.
  Fix    : Remove sample_id from the SELECT in _getLog() or add it to the
           logSessionEvent() INSERT.

BUG-008  stop() sets _running = false AFTER pool.end()
  File   : src/engine/IntegrationEngine.js  stop()
  Detail : The sequence is: (1) clear retry timer, (2) disconnect serial,
           (3) reset framer, (4) pool.end(), (5) _running = false.
           If _connectDbWithRetry() wins a retry race between steps 1 and 5,
           it may call _createDbPool() and then _initialiseDbModules() with
           a new pool while the old pool is being closed. _running flag should
           be set to false FIRST to immediately stop the retry loop.
  Fix    : Set `this._running = false` as the first statement in stop().

BUG-009  config.json has analyser_id field that is never read
  File   : config/analysers/access2_config.json  (line 8)
  Detail : `"analyser_id": "ACCESS2"` is defined but no code in the engine
           reads this key. The code uses analyzer_uid everywhere. This is a
           dead config entry that could mislead future developers.
  Fix    : Remove or rename to a comment-only field.

BUG-010  _getResults() allows status=ERROR filter that never matches
  File   : IntegrationAPI.js  _getResults()
  Detail : `const allowed = ['MAPPED', 'UNMAPPED', 'ERROR']` - the ERROR
           value is listed as valid but lis_results rows only ever have
           mapping_status of MAPPED or UNMAPPED (see BUG-002). Any request
           with ?status=ERROR always returns 0 rows with no explanation.
  Fix    : Remove ERROR from the allowed list, or implement a separate error
           status path in ResultWriter.

BUG-011  Empty patientName stored as empty string instead of null
  File   : src/protocol/Access2Parser.js  _parsePatientRecord()
  Detail : `[firstName, lastName].filter(Boolean).join(' ') || lastName`
           When both firstName and lastName are empty strings, the expression
           evaluates to `'' || ''` which is `''` (empty string), not null.
           An empty string is stored in patient_name in the DB rather than NULL.
  Fix    : `[firstName, lastName].filter(Boolean).join(' ') || null`

BUG-012  CORS default is wildcard '*' with no documentation for production
  File   : IntegrationAPI.js  start()
  Detail : `const limsOrigin = process.env.LIMS_ORIGIN || '*'` - any origin
           can call the API if LIMS_ORIGIN is not set. Acceptable for local
           lab network but .env should enforce setting this before go-live.
  Fix    : Log a warning on startup if LIMS_ORIGIN is not set. Consider
           defaulting to 'http://localhost' rather than '*'.
